# 技能系统通用组件文档

---

## 重要提示

> **修改或添加、删除本文档中列出的任何通用组件时，必须同步更新此文档。**
>
> 新增角色技能时，只能使用通用组件或创建新的通用组件，**禁止在 `game.js` 等游戏引擎文件中编写角色耦合代码**。

---

## 一、技能实现原则

### 1.1 使用通用组件

所有技能效果必须通过通用组件实现。通用组件是独立的、可复用的模块，不依赖任何特定角色的 ID 或属性。

```javascript
// 正确：使用通用组件
game._skillManager.recordTrigger(seat, 'Amae_Koromo', 4, { qty });
game._skillManager.wasTriggered(seat, 'Amae_Koromo', 4);

// 错误：在 game.js 中写角色耦合代码
game._koromoSkill2Qty = qty;  // 禁止！
game._aislinnSkill1UsedRows = usedRows;  // 禁止！
```

### 2.3 SkillManager（技能系统主入口）

**文件**: `src/skill/index.js`

SkillManager 是技能系统与游戏引擎交互的唯一接口，所有技能相关操作必须通过它完成。

#### 2.3.1 回合内触发记录（per-turn）

跨技能联动的基础设施，每次摸牌回合开始时清零。

| 方法 | 签名 | 说明 |
|------|------|------|
| `recordTrigger` | `(seat, characterId, skillIndex, data)` | 记录本回合某技能被触发及附带数据 |
| `wasTriggered` | `(seat, characterId, skillIndex)` → `boolean` | 查询本回合某技能是否被触发过 |
| `getTriggerData` | `(seat, characterId, skillIndex)` → `any` | 获取本回合某技能触发时的附带数据 |
| `clearTurnRecords` | `()` | 清除本回合所有触发记录（每次摸牌前调用） |

#### 2.3.2 跨回合持久数据（per-hand）

技能在一局中存储跨回合的数据，每局开始时清零。

| 方法 | 签名 | 说明 |
|------|------|------|
| `setSkillData` | `(seat, characterId, skillIndex, value)` | 存储技能持久数据 |
| `getSkillData` | `(seat, characterId, skillIndex)` → `any` | 读取技能持久数据 |
| `clearHandData` | `()` | 清除本局所有技能数据（每局开始时调用） |

#### 2.3.3 shouldAutoExecute

可选技能（`isOptional: true`）可以在技能定义中提供 `shouldAutoExecute(context)` 函数。当返回 `true` 时，SkillManager 将该技能视为自动执行，跳过玩家确认环节。

此机制与 `recordTrigger`/`wasTriggered` 配合，可实现"技能 A 触发后，技能 B 自动执行且复用 A 的数据"的模式。

**使用示例（skill-registry.js）**：
```javascript
// 技能 A（BEFORE_DISCARD）：执行后记录触发
execute: async function(context) {
    // ... 执行逻辑 ...
    game._skillManager.recordTrigger(seat, 'CharacterId', 4, { qty });
    context.done();
},

// 技能 B（DISCARD_SELECTED）：检查技能 A 是否触发过，决定是否自动执行
shouldAutoExecute: function(context) {
    return context.game._skillManager.wasTriggered(context.seat, 'CharacterId', 4);
},
execute: function(context) {
    let triggerData = context.game._skillManager.getTriggerData(seat, 'CharacterId', 4);
    let qty = triggerData?.data?.qty || 1;
    // ... 复用技能 A 的选择数据 ...
},
```

### 2.7 牌堆结构

牌山为环形 68 墩（每墩 1 上 1 下），不进行 splice，指针管理所有位置。

#### 区域划分

```
_cursor → 活牌区 → 海底(_haitei-1) → [_haitei]王牌尾部 → 王牌区(_dw_count墩) → 王牌头部(岭上) → 空墩区 → 回到_cursor
```

| 区域 | 说明 |
|------|------|
| 牌山（活牌区） | 可摸牌范围，从 `_cursor` 到海底牌，摸牌方向：上层 → 下层 → 下一墩 |
| 海底 | 牌山最后一墩（`_haitei - 1`），最后一张称为海底牌 |
| 王牌 | 初始为牌堆的最后 7 墩，王牌顺序与牌山相反。开杠后摸取一张岭上牌，并将海底牌移动至王牌尾部形成补充墩 |

#### 王牌区内部布局（从王牌尾部向王牌头部）

| 位置 | 内容 |
|------|------|
| 补充墩 | 海底补充到王牌的墩（最多 2 墩），动态增减，`_dw_count` 随之变化 |
| 宝 5 | 王牌第 7 墩，上方为第 5 张表宝牌指示牌，下方为第 5 张里宝牌 |
| 宝 4 | 王牌第 6 墩，上方为第 4 张表宝牌指示牌，下方为第 4 张里宝牌 |
| 宝 3 | 王牌第 5 墩，上方为第 3 张表宝牌指示牌，下方为第 3 张里宝牌 |
| 宝 2 | 王牌第 4 墩，上方为第 2 张表宝牌指示牌，下方为第 2 张里宝牌 |
| 宝 1 | 王牌第 3 墩，上方为初始宝牌指示牌，下方为第 1 张里宝牌指示牌 |
| 岭 2 | 王牌第 2 墩，岭上牌（上下层各 1 张） |
| 岭 1 | 王牌第 1 墩（王牌头部），岭上牌（上下层各 1 张），杠后最先摸取 |

#### 指针系统（`src/core/shan.js`）

| 指针 | 说明 |
|------|------|
| `_cursor` | 摸牌位置指针（开门位，活牌区头部） |
| `_haitei` | 王牌尾部指针（海底+1，活牌区与王牌区的边界）。**注意：海底牌位于 `_haitei - 1`** |
| `_dw_count` | 王牌区实际墩数（初始 7，补充墩增减时变化） |
| `_dora_start` | 首张宝牌（宝 5）指针，初始 = `_haitei` |
| `_rinshan_drawn` | 岭上已摸次数（0~4） |

#### 王牌移位

牌山定长 68 墩，不 splice。增删牌时通过整体移位实现：

| 方法 | 说明 |
|------|------|
| `shan._shiftDeadWallTowardCursor(n)` | 王牌整体向海底方向移动 n 墩，填满空墩，`_haitei` 和 `_dora_start` 同步减 n |
| `shan._shiftDeadWallAwayFromCursor(n)` | 王牌整体远离海底方向移动 n 墩，空出新墩，`_haitei` 和 `_dora_start` 同步加 n |

注意：移位跨越环形边界（68 ↔ 0）时通过 `% len` 和 `+ len` 兜底正确处理。

### 2.9-3 VIEW_AS_WIN_TILE 完整模式（huleExpander + execute + 选牌 UI）

当技能需要「将不直接构成和牌的牌视为和牌」时，使用 `EffectType.VIEW_AS_WIN_TILE` 搭配以下两件套：

| 组件 | 说明 |
|------|------|
| `huleExpander(context)` | 返回候选牌列表，扩大 `allow_hule` 的判定范围（CHECK_HULE 时点） |
| `execute(context)` | 和牌时由 `_huleExpanderUsed` 路径调用，带 `input` 桥接，支持选牌 UI |

**流程**：

1. `allow_hule` 阶段：huleExpander 返回候补牌 → 扩容器循环用听牌逐一测试 → 成功后设置 `_huleExpanderUsed`
2. `hule()` 阶段：检测到 `_huleExpanderUsed` → 调用 `skill.effect.execute(ctx)`（带 `input.askTileOptions` + `context.done`）
3. 单听牌：自动选牌 → 荣和设 `_overrideRongpai`，自摸调 `winTileOverrider.override` → `done()`
4. 多听牌：弹窗 `input.askTileOptions(candidates, title, defaultFn)` → 玩家选牌 → 同样设置 → `done()`

**重要：execute 必须加 input 守卫**

`hule()` 中 DECLARE_HULE 时点会触发**两次** execute：

| 触发路径 | 位置 | 有 `input` 桥接 | 有 `context.done` |
|----------|------|:---:|:---:|
| `_skill_trigger` auto-execute | game.js:1567 | ❌ | ❌ |
| `_huleExpanderUsed` 路径 | game.js:1788 | ✅ | ✅ |

因此所有 `VIEW_AS_WIN_TILE` 技能的 `execute` 开头**必须**加守卫，只让有桥接的路径执行：

```javascript
execute: async function(context) {
    // 顶层 execute 已在 _huleExpanderUsed 路径被调用（带 input 桥接），
    // 此 auto-execute 路径无条件弹窗 → 跳过
    if (!context.input || typeof context.input.askTileOptions !== 'function') return;
    // ... 选牌逻辑
}
```

无论技能本身是 `DECLARE_HULE` 时点的顶层 execute，还是 `CONTINUOUS` 时点下 `parts` 中的子技能，此守卫规则通用。

**使用此模式的技能**：

| 角色 | 技能 | 触发条件 |
|------|------|----------|
| 天江衣① | 海底/河底视为和牌 | `model.shan.paishu === 0` |
| Test_Character③ | 字牌视为和牌 | 听牌非字牌时，字牌可视为和牌 |
| 姊带丰音④ | 追立/裸单骑视为和牌 | 追立 或 手牌仅剩 1 枚（裸单骑） |

### 2.11 手牌标记（明牌）

**文件**: `src/core/shoupai.js`（核心模型）、`src/ui/shoupai.js`（UI 渲染）

技能可将玩家手牌中的部分牌张「标记」为明牌，被标记的牌张**对所有玩家可见**。

#### 核心 API

| 方法 | 签名 | 说明 |
|------|------|------|
| `markTiles` | `(tiles: string[])` | 标记指定牌张为明牌（`tiles` 如 `['m1','p5','z7']`），重复标记无影响 |
| `unmarkTiles` | `(tiles: string[])` | 取消指定牌张的标记 |
| `clearMarkedTiles` | `()` | 清除该手牌的所有标记 |
| `markedTiles` | getter → `Set<string>` | 获取当前已标记的牌张集合 |

#### 数据同步

标记牌张通过游戏消息自动同步给所有玩家：

- **`zimo` / `dapai` / `fulou` / `gang` 消息**中均携带 `markedTiles` 字段（始终发送，空时为 `[]`）
- 服务端（`src/core/game.js`）从 `model.shoupai[seat].markedTiles` 取值
- 客户端（`src/core/board.js`）在对应的 `zimo()`/`dapai()`/`fulou()`/`gang()` 方法中更新 `_markedTiles`
- 技能无需手动发送网络消息，只需调用 `shoupai.markTiles(tiles)`，后续游戏消息会自动携带

#### UI 渲染效果

| 视角 | 效果 |
|------|------|
| **对手视角** | 被标记牌以**正面牌**展示在 `<span class="marked-tiles">` 容器内，位于手牌**最左侧**；其余牌为暗牌（背面） |
| **自家视角** | 被标记牌带有金色内边框高亮（`.pai.marked` CSS 类），与普通牌区分 |

> **注意**：标记牌与「本巡摸牌」互不干涉。对手视角下，被标记的摸牌仍停留在手牌最右侧的摸牌位置，以**明牌**展示；未被标记的摸牌仍为暗牌。

#### 使用示例

```javascript
// BEFORE_DISCARD 时点：标记手牌中所有宝牌为明牌
execute: function(context) {
    let game = context.game;
    let seat = context.seat;
    let model = game._model;
    let shoupai = model.shoupai[seat];
    if (!shoupai) return { executed: true };

    // 收集手牌中的宝牌
    let baopai = model.shan.baopai;
    let doraSet = tileUtils.buildDoraSet(baopai);
    let handTiles = shoupai.toString().match(/[mpsz]\d/g) || [];
    let doraTiles = handTiles.filter(p => doraSet.has(p));

    // 标记宝牌为明牌
    if (doraTiles.length > 0) {
        shoupai.markTiles(doraTiles);
    }
    return { executed: true };
},

// AFTER_DISCARD 时点：清除标记
execute: function(context) {
    let game = context.game;
    let seat = context.seat;
    let shoupai = game._model.shoupai[seat];
    if (shoupai) shoupai.clearMarkedTiles();
    return { executed: true };
},
```

> **警告**：`markTiles` 传入不存在的牌张或已打出的牌张不会报错但无效。标记会随着玩家打出/副露/杠牌**自动清理**消耗掉的牌张标记（`dapai()`、`fulou()`、`gang()` 中调用 `_markedTiles.delete(p)`）。

> **⚠️ 重要：pickHandTiles / pickHandTilesRange 的 UI 刷新规则**
>
> `pickHandTiles` 和 `pickHandTilesRange` 同时维护**两个 view**：
> 1. 手牌上的序号 badge（通过 `_updateHandBadges`）
> 2. 提示面板中的文字和按钮状态
>
> 修改这两个方法时，**必须通过 `_refreshHandPrompt(selectedOrder, textHtml, btnText, btnEnabled)` 统一刷新两个 view**，禁止在 `updateUI` 中单独调用 `_updateHandBadges()` 后手动更新提示 UI。否则容易出现只刷新手牌 badge 而遗漏提示文字/按钮状态，导致两个 view 不一致的 bug。
>
> ```javascript
> // 正确：通过 _refreshHandPrompt 统一刷新
> let updateUI = () => {
>     that._refreshHandPrompt(selectedOrder,
>         (description || '选择手牌') + ' （已选 ' + selectedOrder.length + '/' + count + '）'
>             + (isMulti ? '<span class="skill-timeout"></span>' : ''),
>         '确定（' + selectedOrder.length + '/' + count + '张）',
>         selectedOrder.length === count
>     );
> };
>
> // 错误：单独调用 _updateHandBadges 后手动更新 UI（容易遗漏）
> let updateUI = () => {
>     that._updateHandBadges(selectedOrder);           // 只刷了 badge
>     that._node.find('.skill-prompt-text').html(...); // 可能遗漏
>     // ... 按钮状态也可能遗漏
> };
> ```

## 三、维护规则

### 3.1 修改组件时

- 修改任何通用组件的 API（方法签名、参数、返回值）后，**必须同步更新本文档中对应的方法签名和说明**
- 如果修改影响了已有技能的行为，必须检查所有引用该组件的技能定义

### 3.2 添加组件时

- 在对应的模块文件中添加新组件
- 在本文档的「二、通用组件清单」中添加新条目，包含：文件路径、方法签名、说明
- 确保新组件不依赖任何角色 ID，通过参数接受所有角色相关信息

### 3.3 删除组件时

- 从本文档中移除对应条目
- 检查所有技能定义，确保没有仍在引用被删除组件的代码

### 3.4 新增角色技能时

- 只能在 `src/skill/skill-registry.js` 的 `SKILL_EXECUTE_MAP` 中添加
- 只能使用本文档列出的通用组件
- 如果需要新的通用能力，先创建通用组件并更新本文档，再在技能中使用
- **禁止在 `game.js` 中添加任何角色专属字段或逻辑**

### 3.5 禁止事项

- 在 `game.js` 中出现任何角色 ID 字符串（如 `'Amae_Koromo'`、`'Aislinn_Wishart'`）
- 在 `game.js` 中出现 `_xxxSkillXQty`、`_xxxSkillXUsedRows` 等角色专属字段
- 直接在 `game.js` 中写角色条件判断（`if (charId === '...')`）
- 技能之间直接通过 game.js 字段传递数据（应使用 `recordTrigger`/`getTriggerData` 或 `setSkillData`/`getSkillData`）

### 4.5 和牌番数流水线架构

和牌相关的技能操作横跨两个阶段，遵循以下流水线：

```
CHECK_HULE 阶段（allow_hule() 内，回答"能不能和"）
│
├── 标准役判定（Majiang 内置）→ 成功则直接通过
│
├── huleExpander（已有）    — 扩大"哪张牌能荣和"的范围（牌层面）
│   └── 返回候选牌列表 → game.js 逐张模拟 _check → 命中则 _huleExpanderUsed
│
├── yakuExpander（新增）    — 扩大"有没有番"的范围（番层面，0番救活）
│   └── 返回虚拟役 [{ name, fanshu }] → 命中则 _yakuExpanderUsed
│
└── huleRestrictor（新增）  — 施加番缚/禁和限制（番层面，有条件禁止）
    └── 返回 { minFan, forbidRon, forbidTsumo } 或 null

_continueHule 内（结算阶段，回答"多少番"）
│
├── Majiang.Util.hule() 计算原始番数
│
├── yakuExpander 虚拟役合并到 hule  ← 在 HULE_SETTLE 之前合并，以便 FanModifier 可见
│
├── HULE_SETTLE 阶段
│   └── FanModifier / hanOps    — 精确追加番数、修改役种价值（数值精确化）
│       ├── setYakuHan(hule, '役名', 新番数)   — 修改已有役种番数
│       ├── addHan(hule, 番数, '名称')         — 添加通用加番（技能番）
│       └── addYakuHan(hule, '役名', delta)    — 增减已有役种番数
│
├── huleRestrictor minFan 精确校验（仅计算 type==='yaku' 的有役番）
│
└── AFTER_HULE
```

**关键设计原则**：
- `yakuExpander` 只管"能不能和"（追加役种），不管番数多少。返回虚拟役的 `fanshu` 为名义值（通常为 1）。
- `FanModifier` / `hanOps` 只管"番数多少"（数值调整），不管能不能和。运行在 HULE_SETTLE 时点，此时 yakuExpander 的虚拟役已合并完成，可直接操作。
- 同一个角色的两个技能可以通过这种时分复用的方式协作：技能 A（yakuExpander）追加役种，技能 B（HULE_SETTLE）调整役种番数。

#### yakuExpander：0番手牌兜底起和

在 CHECK_HULE 阶段，若标准役判定为 0 番，`allow_hule()` 会调用 `getYakuExpanders()` 兜底。

**yakuExpander 上下文字段**：

| 字段 | 来源 | 说明 |
|------|------|------|
| `shoupai` | `model.shoupai[seat]` | 手牌（自摸时已含和了牌；荣和时不含） |
| `rongpai` | 荣和场景传入 | 荣和牌（自摸时为 `undefined`） |
| `game` | 固定 | 游戏引擎实例 |
| `seat` | `getYakuExpanders` 注入 | 技能持有者席位 |
| `player` | 固定 | 当前回合玩家 |

**示例 1：允许有一个刻子的平和（原村和②）**

> 手牌中允许有一个暗刻，除去刻子后若剩余牌型满足非役牌雀头 + 顺子 + 两面听牌，则可计平和。

```javascript
{
    /* yakuExpander：在 CHECK_HULE 阶段兜底，返回虚拟平和役 */
    timing: TimingPoints.CHECK_HULE,
    type: SkillType.PASSIVE,
    effectType: EffectType.MODIFY_YAKU_VALUE,
    yakuExpander: function(context) {
        let shoupai = context.shoupai;
        let rongpai = context.rongpai;
        let game = context.game;
        let seat = context.seat;
        if (!shoupai || !game) return [];

        let model = game._model;
        if (!model) return [];

        let mianziList = Majiang.Util.hule_mianzi(shoupai, rongpai || null);
        if (!mianziList || mianziList.length === 0) return [];

        let zhuangfeng = model.zhuangfeng;
        let menfeng = seat;

        for (let mianzi of mianziList) {
            if (mianzi.length !== 5) continue;

            /* 统计非副露刻子数：恰好1个暗刻 */
            let koutsuCount = 0;
            let hasOpenMeld = false;
            for (let i = 1; i < mianzi.length; i++) {
                let m = mianzi[i];
                if (m.match(/[\+\=\-]/)) { hasOpenMeld = true; break; }
                if (/^[mpsz](\d)\1\1/.test(m)) koutsuCount++;
            }
            if (hasOpenMeld || koutsuCount !== 1) continue;

            /* 检查雀头非役牌（三元牌、场风、自风） */
            let pair = mianzi[0];
            if (/^z[567]/.test(pair)) continue;
            if (pair.startsWith('z' + (zhuangfeng + 1))) continue;
            if (pair.startsWith('z' + (menfeng + 1))) continue;

            /* 检查听牌为两面（非单骑/嵌张/边张） */
            let mWithHulepai = mianzi.find(m => m.includes('!'));
            if (mWithHulepai) {
                if (/^[mpsz](\d)\1[\+\=\-\_]\!/.test(mWithHulepai)) continue;
                if (/^[mps]\d\d[\+\=\-\_]\!\d/.test(mWithHulepai)) continue;
                if (/^[mps](123[\+\=\-\_]\!|7[\+\=\-\_]\!89)/.test(mWithHulepai)) continue;
            }

            /* 满足条件，返回虚拟平和役（名义 1 番，后续由 FanModifier 调整为实际番数） */
            return [{ name: '平和', fanshu: 1 }];
        }

        return [];
    },
}
```

> **注意**：`yakuExpander` 返回的虚拟役 `fanshu` 只是名义值。游戏引擎会将其合并到 `hule.hupai` 并使 `hule.fanshu > 0`，从而允许和牌。真正的番数调整由 HULE_SETTLE 阶段的技能负责。

#### FanModifier / hanOps：HULE_SETTLE 阶段调整番数

在 HULE_SETTLE 时点，`context.hule` 已包含完整的役种列表（含 yakuExpander 合并的虚拟役），可以直接修改番数。

**HULE_SETTLE 上下文字段**：

| 字段 | 说明 |
|------|------|
| `hule` | 和牌计算结果 `{ hupai: [{name, fanshu, type}], fu, fanshu, defen, fenpei }`。`type` 取值：`'yaku'`（有役番）、`'dora'`（宝牌）、`'bonus'`（无役奖励番） |
| `shoupai` | 和牌家手牌 |
| `rongpai` | 荣和牌（自摸时为 `null`） |
| `game` | 游戏引擎实例 |
| `seat` | 技能持有者席位 |

**hanOps API 速查**（`src/effect/han-ops.js`）：

| 函数 | 说明 |
|------|------|
| `setYakuHan(hule, '役名', 新番数)` | 设置指定役种的番数（不存在则新增，追加 `type: 'yaku'`） |
| `addHan(hule, 番数, '名称')` | 添加通用加番（追加 `type: 'bonus'` 的无役奖励番到 hupai 末尾） |
| `addYakuHan(hule, '役名', delta)` | 增减已有役种的番数（可为负） |
| `removeYaku(hule, '役名')` | 移除指定役种 |
| `hasYaku(hule, '役名')` | 检查是否有某役种 |
| `getTotalHan(hule)` | 获取总番数 |

**示例 2：断幺/平和视为2番役（原村和①）**

> 这是 yakuExpander（技能②）的配套技能。技能②追加平和役后，由本技能统一提升断幺九和平和至 2 番。

```javascript
{
    /* HULE_SETTLE：将已有的断幺九/平和提升至 2 番 */
    timing: TimingPoints.HULE_SETTLE,
    type: SkillType.PASSIVE,
    effectType: EffectType.MODIFY_YAKU_VALUE,
    condition: function(context) {
        return !!(context.hule && context.hule.hupai);
    },
    execute: function(context) {
        let hule = context.hule;
        if (hanOps.hasYaku(hule, '断幺九')) {
            hanOps.setYakuHan(hule, '断幺九', 2);
        }
        if (hanOps.hasYaku(hule, '平和')) {
            hanOps.setYakuHan(hule, '平和', 2);
        }
        return { executed: true };
    },
}
```

> **关键**：技能②（yakuExpander）在 CHECK_HULE 阶段追加平和，其虚拟役在 HULE_SETTLE 之前被合并到 `hule.hupai`。因此技能①（HULE_SETTLE）的 `hasYaku(hule, '平和')` 可以看到此役并提升至 2 番。两个技能完全独立，无需优先级调度。

**示例 3：按条件追加技能番**

```javascript
{
    timing: TimingPoints.HULE_SETTLE,
    type: SkillType.PASSIVE,
    effectType: EffectType.MODIFY_YAKU_VALUE,

```

#### huleRestrictor：向他人施加番缚/禁和

huleRestrictor **不属于和牌家**，而是由其他玩家的技能施加限制。SkillManager 会遍历所有其他座位的技能。

> **重要**：番缚（`minFan`）只计算**有役番**（`type === 'yaku'`），不计算宝牌（`type: 'dora'`）和无役奖励番（`type: 'bonus'`，即 `addHan` 追加的技能番）。

```javascript
// 番缚示例：高鸭稳乃——牌河 N 张牌后起和需 M 番
{
    huleRestrictor: function(context) {
        let model = context.game._model;
        let target = context.seat;  // 被检查的潜在和牌家
        let riverSize = model.he[target]._pai.filter(
            p => !p.match(/[\+\=\-]$/) && p.slice(-1) !== '_'
        ).length;
        if (riverSize >= 16) return { minFan: 4 };
        if (riverSize >= 12) return { minFan: 3 };
        if (riverSize >= 6)  return { minFan: 2 };
        return null;  // 无限制
    },
}

// 此巡禁和示例：狩宿巴——打出白板后此巡其他玩家不能荣和
{
    // execute 中记录标记
    execute: function(context) {
        context.game._skillManager.recordTrigger(
            context.seat, 'Karijuku_Tomoe', 2, { blocked: true });
        return { executed: true };
    },
    // huleRestrictor 检查标记
    huleRestrictor: function(context) {
        let sm = context.game._skillManager;
        let mySeat = context.mySeat;  // 限制施加者
        if (sm.wasTriggered(mySeat, 'Karijuku_Tomoe', 2)) {
            if (context.seat !== mySeat) {
                return { forbidRon: true, forbidTsumo: true };
            }
        }
        return null;
    },
}
```

> **重要**：`huleRestrictor` 的 context 中会额外注入 `mySeat`（限制施加者的席位），
> 与 `context.seat`（被限制的潜在和牌家）区分。

### 4.6 parts 子技能模式（多时点技能）

当一个技能需要在**多个不同时点**触发不同效果时，使用 `parts` 数组将技能拆分为多个**子技能**。每个子技能拥有独立的 `timing`、`condition` 和 `execute`，技能管理器在对应时点分别触发它们。

**核心结构**：

```javascript
技能索引: {
    timing: TimingPoints.CONTINUOUS,     // 主技能时点（CONTINUOUS 表示由 parts 接管触发）
    type: SkillType.PASSIVE,
    effectType: EffectType.XXX,          // 主效果类型
    parts: [
        {
            /* Part A */
            timing: TimingPoints.DISCARD_SELECTED,  // 子技能 A 的触发时点
            condition: function(context) { ... },    // 子技能 A 的触发条件
            execute: function(context) { ... },      // 子技能 A 的执行逻辑
        },
        {
            /* Part B */
            timing: TimingPoints.HULE_SETTLE,
            condition: function(context) { ... },
            execute: function(context) { ... },
        },
        {
            /* Part C */
            timing: TimingPoints.FENPEI_CALCULATED,
            condition: function(context) { ... },
            execute: function(context) { ... },
        },
    ],
},
```

**关键规则**：
- 主技能 `timing` 通常设为 `CONTINUOUS`，表示不直接在单一固定时点触发
- 每个 part 必须有独立的 `timing` 字段（覆盖主技能的 timing）
- `condition`、`execute` 均独立，各自判断和执行
- `usageType`/`usageMax` 由主技能统一管理（parts 共享使用次数）
- `priority`、`effectType` 也可由 part 独立覆盖

#### 示例：姊带丰音 ② — 追立时立直改为2番 / 铳点减半

此技能需要横跨三个时点完成三个独立操作，天然适合 `parts` 模式：

```
Part A — DISCARD_SELECTED（舍牌时点）
  ↓ 检测到追立（其他玩家已立直时自己宣言立直）
  ↓ 记录标记：_chaseRiichi = true（通过 setSkillData）
  
Part B — HULE_SETTLE（和牌番数计算时点）
  ↓ 检查 _chaseRiichi 标记 + 手牌有「立直」役
  ↓ 将「立直」役番数从 1 改为 2（setYakuHan）
  
Part C — FENPEI_CALCULATED（点数分配计算时点）
  ↓ 自己是放铳方（wasRonned）且满足条件（追立 或 3副露以上）
  ↓ 将己方支付点数减半（multiplyPayment × 0.5，百位向下取整）
  ↓ 同步减少和牌家收入（收支平衡）
```

**完整代码**：

```javascript
'Anetai_Toyone': {
    1: {
        /* ② 追立时立直改为2番 / 铳点减半 — 三个子技能分别在不同时点触发 */
        timing: TimingPoints.CONTINUOUS,
        type: SkillType.PASSIVE,
        effectType: EffectType.MODIFY_YAKU_VALUE,

        parts: [
            {
                /* Part A — DISCARD_SELECTED：记录是否追立 */
                timing: TimingPoints.DISCARD_SELECTED,
                condition: function(context) {
                    let game = context.game;
                    if (!game || context.player !== context.seat) return false;
                    /* 仅当打出的牌带 * 后缀（宣言立直）时触发 */
                    return !!(context.dapai && context.dapai.slice(-1) === '*');
                },
                execute: function(context) {
                    let game = context.game;
                    let seat = context.seat;
                    /* 检查是否有其他玩家已立直 → 追立 */
                    for (let l = 0; l < 4; l++) {
                        if (l !== seat && game._lizhi && game._lizhi[l]) {
                            game._skillManager.setSkillData(
                                seat, 'Anetai_Toyone', '_chaseRiichi', true);
                            break;
                        }
                    }
                    return { executed: true };
                },
            },
            {
                /* Part B — HULE_SETTLE：追立时立直改为2番 */
                timing: TimingPoints.HULE_SETTLE,
                condition: function(context) {
                    return context.seat === context.player
                        && !!(context.hule && context.hule.hupai);
                },
                execute: function(context) {
                    let hule = context.hule;
                    let seat = context.seat;
                    let game = context.game;
                    let isChaseRiichi = game._skillManager
                        ? game._skillManager.getSkillData(seat, 'Anetai_Toyone', '_chaseRiichi')
                        : false;
                    if (isChaseRiichi && hanOps.hasYaku(hule, '立直')) {
                        hanOps.setYakuHan(hule, '立直', 2);
                    }
                    return { executed: true };
                },
            },
            {
                /* Part C — FENPEI_CALCULATED：铳点减半 */
                timing: TimingPoints.FENPEI_CALCULATED,
                condition: function(context) {
                    return context.seat === context.player
                        && !!(context.hule && context.hule.fenpei);
                },
                execute: function(context) {
                    let hule = context.hule;
                    let seat = context.seat;
                    let game = context.game;
                    let isChaseRiichi = game._skillManager
                        ? game._skillManager.getSkillData(seat, 'Anetai_Toyone', '_chaseRiichi')
                        : false;
                    let stats = fanModifier.getMeldStats(game._model, seat);
                    let nonAnkanMelds = stats.total - stats.ankan;
                    /* 放铳方 且 (追立 或 3副露以上) → 铳点减半 */
                    if (context.wasRonned && (isChaseRiichi || nonAnkanMelds >= 3)) {
                        let oldPayment = hule.fenpei[seat];
                        pointPayment.multiplyPayment(hule.fenpei, seat, 0.5);
                        let newPayment = hule.fenpei[seat];
                        /* 铳点减半后，和牌家收入同步减少 */
                        let winnerSeat = context.ronBy;
                        if (winnerSeat != null && winnerSeat !== seat) {
                            let diff = oldPayment - newPayment;
                            hule.fenpei[winnerSeat] += diff;
                        }
                    }
                    return { executed: true };
                },
            },
        ],
    },
}
```

**Part 间数据传递**：

Part A 在 DISCARD_SELECTED 时记录 `_chaseRiichi` 标记，Part B 和 Part C 在后续时点通过 `getSkillData` 读取该标记。此模式使用 SkillManager 的跨回合持久数据（`setSkillData`/`getSkillData`），在一局中持续有效。

**铳点减半取整规则**：

`multiplyPayment` 对支付点数 × 0.5 后，以 100 为单位向下取整（绝对值）。例如：-1500 × 0.5 = -750 → 取绝对值 750 → `floor(750/100)*100` = 700 → 最终 -700。减半后同步减少和牌家的收入（`diff = 旧支付 - 新支付`），保证总分收支平衡。

**适用场景**：

当一个技能的效果分散在游戏的不同阶段，且每个阶段的操作独立、不依赖用户输入时，使用 `parts` 比拆分为多个独立技能更清晰。例如：
- 舍牌时记录标记 → 结算时读取标记执行效果
- 出牌前修改规则 → 和牌时应用修改

如果各阶段需要用户交互（如选牌），应考虑拆分为独立技能并使用 `recordTrigger`/`wasTriggered` 模式（见 4.3 节）。

### 5.1 技能定义结构

每个技能在 `src/skill/skill-registry.js` 的 `SKILL_EXECUTE_MAP` 中定义，按角色 ID 和技能索引组织，结构如下：

```javascript
'角色ID': {
    技能索引: {
        /* ─── 触发与类型（必填） ─── */
        timing: TimingPoints.XXX,   // 触发时机
        type: SkillType.XXX,        // PASSIVE / ACTIVE / CONDITIONAL
        isOptional: true/false,     // 是否需要玩家确认（CONDITIONAL 通常为 true）

        /* ─── 优先级（可选，默认 100） ─── */
        priority: 300,

        /* ─── 使用限制（可选，默认 UNLIMITED） ─── */
        usageType: UsageType.XX,
        usageMax: N,

        /* ─── 效果类型（可选） ─── */
        effectType: EffectType.XX,

        /* ─── 触发条件（必填） ─── */
        condition: function(context) {
            // 返回 true 表示符合触发条件
        },

        /* ─── 效果执行（必填） ─── */
        execute: function(context) {       // 同步技能
            return { executed: true };
        },
        // 或
        execute: async function(context) { // 异步技能（需要玩家输入）
            // 通过 await input.askXxx() 获取输入
            // 完成后调用 context.done()
        },

        /* ─── 可选：技能间联动 ─── */
        shouldAutoExecute: function(context) {
            // 返回 true 时跳过玩家确认，直接自动执行
        },

        /* ── 可选：范围扩展器 ── */
        huleExpander: function(context) { return [...]; },
        tenpaiExpander: function(context) { return [...]; },
        ponExpander: function(context) { return [...]; },
        kanExpander: function(context) { return [...]; },
        chiExpander: function(context) { return [...]; },
        yakuExpander: function(context) { return [{ name: '役名', fanshu: 1 }]; },
        huleRestrictor: function(context) { return null; },

        /* ─── AI 决策（isOptional 为 true 时必填） ─── */
        aiDecision: function(context) {
            // 返回 boolean 或 { activate: boolean, choice: {...} }
        },
    },

```

### 5.2 context 上下文对象

技能的函数（`condition`、`execute`、`aiDecision`、`shouldAutoExecute`、扩展器）都会收到 `context` 对象。不同时点注入的字段不同。

#### 通用字段（所有时点都有）

| 字段 | 类型 | 说明 |
|------|------|------|
| `game` | `Game` | 游戏引擎实例。通过 `game._model` 访问牌局数据，`game._skillManager` 访问技能管理器 |
| `seat` | `number` | 技能持有者的游戏席位（0=东/1=南/2=西/3=北） |
| `player` | `number` | 当前回合玩家的游戏席位 |

#### 特定时点字段

| 时点 | 额外字段 | 说明 |
|------|------|------|
| `BEFORE_DRAW` | `turnType`, `isExtraTurn`, `isRiichi`, `isFirstTurn`, `isDealerFirst` | 摸牌前上下文 |
| `DRAW_SOURCE` | `sourceType`, `isExtraTurn` | 摸牌来源 |
| `BEFORE_DISCARD` | `dapai` | 即将打出的牌字符串（如 `'m1_'`） |
| `DISCARD_SELECTED` | `dapai` | 已选定的舍牌 |
| `AFTER_DISCARD` | `dapai` | 刚刚打出的牌 |
| `DECLARE_HULE` | `shoupai`, `rongpai`, `lunban` | 和牌宣言上下文 |
| `HULE_SETTLE` | `huleInfo` | 和牌结算信息 |
| `CHECK_HULE` / `CHECK_TENPAI` | `shoupai` | 手牌对象（Majiang.Shoupai 实例） |
| `HULE_SETTLE` | `hule` | 和牌计算结果 `{ hupai, fu, fanshu, defen, fenpei }`，可被技能修改 |
| `HULE_SETTLE` | `shoupai`, `rongpai` | 和牌家手牌和荣和牌 |
| `HULE_SETTLE`（放铳者） | `wasRonned: true`, `ronBy` | 标记为放铳者，`ronBy` 为和牌家席位 |

#### execute 独有的字段

`execute` 函数在 `_executeOptionalSkill` 中被调用时，额外注入：

| 字段 | 类型 | 说明 |
|------|------|------|
| `input` | `Object` | 交互接口（见 5.3），提供 `askConfirm`/`askNumber`/`askTextOptions`/`askTileOptions` 四个方法 |
| `done` | `Function` | 技能执行完毕回调。**异步技能必须调用 `context.done()`，同步技能不需要** |

### 5.3 input 交互接口

`context.input` 提供四个 async 方法，在技能执行过程中向玩家索取输入。每个方法同时支持 AI 和人类：

```javascript
// 每个 input 方法的最后一个参数是 aiPicker（AI 回退函数）
// AI 玩家调用时直接执行 aiPicker() 返回结果，不弹 UI
// 人类玩家调用时弹出 UI 等待输入
```

| 方法 | 参数 | 返回值 | 说明 |
|------|------|------|------|
| `askConfirm(desc, aiPicker)` | `desc`：确认提示文本<br>`aiPicker`：`() => 'yes'\|'no'` | `Promise<'yes'\|'no'>` | 询问是否发动技能 |
| `askNumber(min, max, desc, aiPicker)` | `min/max`：数字范围<br>`desc`：提示文本<br>`aiPicker`：`() => number` | `Promise<number\|null>` | 选择数字，null 表示取消 |
| `askTextOptions(options, values, desc, aiPicker)` | `options`：显示文本数组<br>`values`：选项值数组<br>`desc`：提示文本<br>`aiPicker`：`() => value` | `Promise<string\|null>` | 选择文字选项，null 表示取消 |
| `askTileOptions(candidates, desc, aiPicker)` | `candidates`：牌字符串数组<br>`desc`：提示文本<br>`aiPicker`：`() => pai` | `Promise<string\|null>` | 选择牌（显示牌图片），null 表示取消 |

**重要**：AI 玩家走 `aiPicker` 回退，不弹 UI。因此每个 `askXxx` 调用都必须提供有意义的 `aiPicker` 参数。

### 5.4 同步技能 vs 异步技能

#### 同步技能

不需要玩家输入时，`execute` 为普通函数，**直接返回 `{ executed: true }`**，不调用 `context.done()`：

```javascript
execute: function(context) {
    let game = context.game;
    let seat = context.seat;
    let model = game._model;

    // 执行纯数据操作
    let moved = tileOps.popEnd(model, 2);
    tileOps.pushDeadWall(model, moved);

    return { executed: true };  // 同步完成，框架立即继续
},
```

#### 异步技能（需要玩家输入）

需要玩家输入时，`execute` 为 `async function`，通过 `await input.askXxx()` 获取输入，**最后必须调用 `context.done()`**：

```javascript
execute: async function(context) {
    let game = context.game;
    let seat = context.seat;
    let input = context.input;

    // 步骤 1：询问数字
    let count = await input.askNumber(1, 5, '选择额外巡目数',
        () => Math.floor(Math.random() * 5) + 1);
    if (count === null) { context.done(); return; }  // 用户取消

    // 步骤 2：询问文字选项
    let choice = await input.askTextOptions(
        ['移除牌山首部', '移除牌山尾部'],
        ['front', 'end'],
        '选择操作',
        () => Math.random() < 0.5 ? 'front' : 'end');
    if (!choice) { context.done(); return; }

    // 步骤 3：询问牌选项
    let pai = await input.askTileOptions(matches,
        '选择一张牌',
        () => matches[0]);
    if (!pai) { context.done(); return; }

    // 执行效果...
    context.done();  // 通知框架执行完毕
},
```

### 5.5 人类 vs AI 分叉流程

技能系统的分叉由框架层自动处理，技能编写者无需关心。完整流程如下：

```
可选技能触发
  ├── AI 玩家
  │     ├── 调用 aiDecision(context) → 返回 true/false (或 { activate } )
  │     ├── true → 标记已使用 → 执行 execute()
  │     │           └── execute 内的 input.askXxx() → 自动调用 aiPicker 回退函数
  │     └── false → 跳过
  │
  └── 人类玩家
        ├── 若为 BEFORE_DISCARD 时点 → UI 按钮触发 → skipConfirm=true
        │     └── 直接 → 标记已使用 → 执行 execute()
        │
        ├── shouldAutoExecute=true → 视为自动执行
        │     └── 直接 → 标记已使用 → 执行 execute()
        │
        └── 其他情况 → skillPrompt.askConfirm() 弹窗确认
              ├── yes → 标记已使用 → 执行 execute()
              │           └── execute 内的 input.askXxx() → 弹出 UI 等待输入
              └── no  → 跳过
```

**技能编写者只需写好 `condition`、`execute`、`aiDecision` 三个函数**。分叉逻辑和 UI 交互由框架完成。

### 5.6 AI 决策编写指南

`aiDecision(context)` 在可选技能触发时被调用，判定 AI 是否发动技能。

#### 方式一：返回 boolean（简单决策）

```javascript
aiDecision: function(context) {
    // 50% 概率发动
    return Math.random() < 0.5;
},
```

#### 方式二：基于向听数的概率决策

```javascript
aiDecision: function(context) {
    let game = context.game;
    let seat = context.seat;
    let model = game._model;
    let shoupai = model.shoupai[seat];
    let xiangting = shoupai ? Majiang.Util.xiangting(shoupai) : 99;

    if (xiangting <= 0) return Math.random() < 0.8;   // 听牌：80%
    if (xiangting === 1) return Math.random() < 0.5;   // 一向听：50%
    if (xiangting === 2) return Math.random() < 0.3;   // 二向听：30%
    return Math.random() < 0.15;                         // 其他：15%
},
```

#### 方式三：返回选择数据 `{ activate, choice }`

某些技能需要 AI 在决策时选定目标（如从牌河选牌），可返回 `{ activate, choice }`：

```javascript
aiDecision: function(context) {
    let riverTiles = context.riverTiles || [];
    let shoupai = context.shoupai;
    let xiangtingFn = context.xiangtingFn;

    if (!shoupai || !xiangtingFn || riverTiles.length === 0) {
        return { activate: false };
    }

    let currentShanten = xiangtingFn(shoupai);
    let bestScore = -Infinity;
    let bestTile = null;

    // 遍历牌河所有候选，选向听改善最大的
    for (let tile of riverTiles) {
        if (tile.seat === context.seat) continue;  // 跳过自己的牌
        let testShoupai = shoupai.clone();
        testShoupai.zimo(tile.pai);
        let newShanten = xiangtingFn(testShoupai);
        let improvement = currentShanten - newShanten;
        let score = improvement * 100 + (tile.pai ? tile.pai.charCodeAt(1) : 0);
        if (score > bestScore) {
            bestScore = score;
            bestTile = tile;
        }
    }

    if (bestScore > 0 && bestTile) {
        return {
            activate: true,
            choice: { pai: bestTile.pai, seat: bestTile.seat, index: bestTile.index }
        };
    }
    return { activate: false };
},
```

#### AI 决策编写要点

1. **防御性检查**：先检查 `context.game`、`_model`、`shoupai` 等是否存在，避免空指针
2. **利用 Majiang.Util.xiangting()**：向听数是评估手牌质量的核心指标
3. **使用概率而非确定性**：避免 AI 行为过于机械可预测
4. **考虑局势**：听牌时积极发动，离听牌远时保守
5. **带选择数据的技能**：`execute` 中要处理人类玩家跳过了 `aiDecision` 的情况，做好 fallback

## 六、FAQ

### Q1: 舍牌前（BEFORE_DISCARD）的技能，人类玩家和 AI 玩家的触发流程有什么区别？

人类玩家和 AI 玩家的分支流程由框架自动处理，具体如下：

- **人类玩家**：回合内，UI 界面会显示可用的 BEFORE_DISCARD 技能按钮。玩家主动点击按钮 → 直接执行技能效果（`skipConfirm=true`），**不弹确认对话框**。这是为了避免"先弹确认窗、点击确认后又要重复选技能"的冗余交互。

- **AI 玩家**：框架调用 `aiDecision(context)` 判断是否发动 → 若返回 `true`，执行技能效果。

- **`shouldAutoExecute` 特例**：如果技能的 `shouldAutoExecute(context)` 返回 `true`（如检测到同角色其他技能已触发），则无论人类还是 AI，都视为自动执行，不弹确认也不走 `aiDecision`。

参见文档 5.5 节的分叉流程图。

### Q2: 暗切（Hidden Discard）的牌有什么规则？

通过技能效果暗切的牌具有以下特性：

- **对其他人不可见**：暗切的牌在牌河中仅对暗切者本人可见，其他玩家的 UI 上不显示牌面信息
- **不可鸣牌**：暗切的牌不能被其他玩家吃（`CHI`）、碰（`PON`）、杠（`KAN`）
- **不可普通荣和**：其他玩家不能以暗切的牌作为荣和牌
- **唯一例外——视为和牌技能**：荣和家如果拥有"将任意牌视为和牌"的技能（通过 `huleExpander` 扩展器 + `winTileOverrider`），则可以荣和暗切的牌。此时的逻辑是：暗切的牌被技能"视为"手牌的听牌候选，而非直接读取暗切牌面的信息

实现上，暗切通过设置 `game._extra_hidden_discard = true` 标记，`dapai()` 方法检测到该标记后，将舍牌状态改为暗置。`allow_hule()` 中会检查牌的可见性来阻挡普通荣和。

### Q3: 字牌能参与手牌-牌河交换吗？

不能。字牌（`z1`~`z7`）没有花色，而交换技能（如新子憧①）的匹配规则为「花色或点数相同」。字牌既无花色可匹配，其点数也无法与非字牌构成有效匹配，因此字牌不可作为交换手牌的选择对象。在技能 `execute` 和 `aiDecision` 中需过滤掉 `suit === 'z'` 的手牌。

### Q4: 牌河牌数 / 一排牌数的统计规则？

统计牌河牌数或一排牌数时遵循以下规则：

1. **不计副露走的牌**：被其他玩家吃/碰/大明杠取走的牌不计入牌河数量。这些牌带有 `+`、`-`、`=` 方向标记，统计时应排除 `pai.match(/[\+\=\-]$/)`。
2. **不计已被技能从牌河中取走的牌**：若技能（如新子憧①的交换、Aislinn 的牌河摸牌等）已从牌河中取走某张牌，该牌已不在 `he._pai` 数组中，自然不计入。
3. **荣和牌不计入完成牌河判定**：荣和时，放铳家打出的那张荣和牌（即最后一张被荣和的舍牌）不计入该放铳家的牌河排数统计。自摸时不受此影响。

### Q5: 幺九牌的定义？

幺九牌指**老头牌**和**字牌**的总称：

1. **老头牌**：任意花色（`m`/`p`/`s`）中数字为 1 或 9 的牌（如 `m1`、`p9`、`s1`）
2. **字牌**：风牌（`z1`~`z4`：东南西北）和三元牌（`z5`~`z7`：白發中）

判定方式：`suit === 'z'` 或者 `number === 1 || number === 9`。
