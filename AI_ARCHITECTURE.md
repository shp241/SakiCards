# 電脳麻将 AI 架构详解

## 概述

AI 系统源自 `@kobalab/majiang-ai v1.1.0`，已本地化到 `src/ai/` 目录。AI 基于手牌评估值（预期打点）做决策，而非深度学习。

**核心思路**：对每种候选操作（打牌/吃/碰/杠/荣和），向前模拟摸→打循环，计算最终得点的加权期望值，选期望值最高的操作。

---

## 一、文件结构

| 文件 | 作用 |
|------|------|
| `ai/index.js` | 入口，导出 `Player` 类 |
| `ai/player.js` | AI 玩家主体：决策逻辑（吃/碰/杠/立直/和了/打牌选择） |
| `ai/suanpai.js` | 算牌引擎：牌山剩余枚数统计、牌价值评估、危险度计算 |
| `ai/minipaipu.js` | 微型牌谱回放器：用于 AI 向前模拟时构造游戏状态 |

## 二、算牌引擎 (SuanPai)

算牌引擎追踪 4 种牌（m/p/s/z）的剩余枚数，并计算牌的价值和危险度。

### 2.1 牌的计数追踪

SuanPai 维护 `this._paishu`：
```javascript
{
    m: [赤数, 1万×4, 2万×4, ..., 9万×4],  // 10个元素
    p: [赤数, 1筒×4, ..., 9筒×4],
    s: [赤数, 1索×4, ..., 9索×4],
    z: [0, 东×4, 南×4, 西×4, 北×4, 白×4, 发×4, 中×4]  // 8个元素
}
```

每个 `同` 事件发生时调用对应方法减少计数：
- `qipai(qipai)`: 扣除自己的起始手牌 + 宝牌指示牌
- `zimo(zimo)`: 扣除自摸牌
- `dapai(dapai)`: 扣除他家打出的牌（暗切不透露牌种，只标记）
- `fulou(fulou)`: 扣除他家吃的牌
- `gang(gang)`: 扣除杠的牌
- `kaigang(kaigang)`: 扣除新的宝牌指示牌

### 2.2 Paishu 类（牌数加权值）

`get_paishu()` 返回的 `Paishu` 对象提供两个方法：
- `val(p)`: 某张牌能摸到的**期望次数** = `剩余枚数 × 剩余摸牌次数 / 剩余牌总数`
- `pop(p)` / `push(p)`: 用于模拟摸牌时临时增减

### 2.3 牌价值 (paijia)

`paijia(p)` 用于打牌排序，值越小的牌越倾向于先打出。计算逻辑：

**字牌**：
- 剩余枚数 × 权重
- 场风/自风/三元牌权重 ×2
- 宝牌权重 ×2

**数牌**：
- 取以该牌为中心的 5 个相邻位置的牌数（构成搭子的潜力）取加权和
- 离中心的距离越近权重越高
- 赤宝牌 ×2

**`make_paijia(shoupai)`** 额外乘以手牌结构系数：
- 字牌速攻系数：手中同一种类字牌（风/三元）≥ 6~9 枚时 ×4~8
- 混一系数：手牌中某色+字牌 ≥ 10 枚时 ×2~4

### 2.4 危险度计算 (suan_weixian)

当有人立直时，`suan_weixian(p, l)` 计算单张牌对玩家 `l` 的危险度：

| 条件 | 危险度 |
|------|--------|
| 该牌已被该玩家舍出 | 0 |
| 剩余 3 枚（字牌） | 8 |
| 剩余 3 枚（数牌） | 3 |
| 剩余 2 枚 | 3 |
| 剩余 1 枚 | 1 |
| 两侧搭子（如持 13 等 2）| 10（边张）/ 3（中张） |
| 嵌张搭子（如持 24 等 3）| 3 |
| 无相关搭子 | 0 |

`make_paijia` 整合所有立直家的危险度，取最大值归一化到 0~100 区间。庄家立直危险度 ×1.5。

## 三、AI 决策流程 (Player)

AI Player 继承自 `Majiang.Player`，通过覆写 `action_*` 方法来拦截游戏事件并做出决策。

### 3.1 事件处理入口

| 事件 | 逻辑 |
|------|------|
| `action_kaiju` | 无条件跳过 |
| `action_qipai` | 无条件跳过 |
| `action_zimo` | 自家摸牌：荣和 → 流局满贯 → 暗杠 → 打牌 |
| `action_dapai` | 他家打牌：自家 → 倒牌/跳过；他家 → 荣和 → 吃碰杠 → 倒牌 → 跳过 |
| `action_fulou` | 自家吃碰后：补打一张 |
| `action_gang` | 他家杠后：抢杠判断 |
| `action_hule/pingju/jieju` | 无条件跳过 |

### 3.2 决策优先级

每个决策点按以下优先级：

1. **荣和 (select_hule)** — 能赢就赢
2. **流局满贯 (select_pingju)** — 摸牌后手牌 ≥ 4 向听 + 规则允许
3. **杠 (select_gang)** — 暗杠/加杠（需不影响向听数或大幅提升期望值）
4. **吃/碰 (select_fulou)** — 需要减少向听数
5. **打牌 (select_dapai)** — 核心决策

### 3.3 打牌选择 (select_dapai) — 核心算法

```
1. 计算安危牌：若有人立直，找出危险度最低的牌作"安全牌"候选
2. 对所有可打牌按 paijia（牌价值）降序排列（价值低的先打）
3. 对每张候选牌：
   a. 模拟打出该牌后的手牌
   b. 调用 eval_shoupai() 计算期望打点
   c. 结合危险度过滤高风险低收益选择
   d. 记录最高期望值的牌
4. 对"退向牌"（向听不降但听牌面好的牌）做 eval_backtrack 再评估
5. 若期望值 ≥ 350 + 可立直 → 打牌并立直
6. 返回最优打牌
```

**危险度阈值**（安全优先）：
| 条件 | 阈值 |
|------|------|
| 危险度 ≥ 13 | 直接跳过 |
| 2向听以上 + 期望 < 80 | 危险度 ≥ 8 跳过 |
| 向听 > 0 + 期望 < 750 | 危险度 ≥ 8 跳过 |
| 听牌 + 期望 < 50 | 危险度 ≥ 8 跳过 |

### 3.4 手牌评估 (eval_shoupai) — 递归模拟

这是 AI 的核心评估函数，递归向下模拟摸→打的循环：

```
eval_shoupai(shoupai, paishu):
  向听数 == -1 (已听牌且摸到和了牌) → 返回 get_defen() 打点
  手中有待打的摸牌 → 尝试每种打牌，取 max
  向听数 < 3 → 遍历所有可听牌：
    for each 听牌候选 p:
      new_shoupai = shoupai.zimo(p)   // 模拟摸到
      递归 eval_shoupai(new_shoupai)
      结果 += ev × paishu.val(p)       // 加权
    结果 /= width[向听数]              // 8/32/256
  向听数 ≥ 3 → 只计算听牌种类数（不做深度模拟）
```

**`width` 数组**：`[8, 32, 256]`，向听数越深，分母越大，模拟精度越低——平衡计算量与准确度。

**缓存**：`_eval_cache` 按手牌字符串缓存结果，避免重复计算。

### 3.5 听牌判断 (xiangting)

`xiangting()` 覆写了 Majiang 标准向听计算，同时考虑 **6 种役的最短向听**：
1. **门前清**：用 `Majiang.Util.xiangting()` 标准计算
2. **役牌**（场风/自风/三元）：统计已有刻子，有对子且牌山有剩余时向听+1
3. **断幺九**：模拟去除所有 1/9/字牌后计算
4. **对对**：统计刻子+对子数
5. **混一色**（m/p/s 各计算一次）
6. 取 6 种的最小值

### 3.6 吃碰选择 (select_fulou)

分为两段：

**向听 < 3（进攻型）**：
- 列出杠/碰/吃所有可能面子
- 对每种副露模拟后的手牌计算 `eval_shoupai()`
- 选期望值最高的副露
- **已立直时额外限制**：向听不降 + ev < 750 或听牌 + ev < 250 则不吃碰

**向听 ≥ 3（防守型）**：
- 只考虑碰/吃（不考虑杠）
- 必须减少向听数才吃碰（用 `xiangting()` 而非标准向听）

### 3.7 杠选择 (select_gang)

**向听 < 3**：模拟每种杠后的手牌，期望值 > 当前最大值则杠。

**向听 ≥ 3**：杠后向听数不变即可杠（为了增加宝牌指示牌）。

## 四、微型牌谱回放 (minipaipu)

`minipaipu(player, baseinfo, heinfo)` 用于将已发生的事件回放到一个干净的 Player 实例上，以获得准确的牌山剩余信息。

### 输入参数

| 参数 | 类型 | 说明 |
|------|------|------|
| `baseinfo` | `{paistr, zhuangfeng, menfeng, baopai, hongpai, xun}` | 基础状态 |
| `heinfo` | `string[4]` | 4 家河信息 + 副露 |
| `fix` | `boolean` | 是否修正他家副露 |

### 执行流程

1. `kaiju()` 初始化游戏
2. `qipai()` 开局配牌
3. `play_heinfo()` 逐巡回放摸→打→副露
4. 补开杠宝牌

### play_heinfo 回放细节

河信息格式：每家的字符串由 `,` 分割
- 第一个元素：舍牌序列（如 `m1m2p3^m4`，`^` 表示暗杠位）
- 后续元素：副露面子的 mianzi 字符串

回放逻辑精确模拟：
- 摸牌 → 打牌 → 副露（吃碰杠）的顺序
- 暗杠/加杠的特殊处理
- 荣和/抢杠的牌列入河

## 五、综合评估示例

假设 AI 在第 6 巡摸到牌，需要决定打什么：

```
1. SuanPai 已追踪: 牌山剩余 50 枚，AI 自己要摸约 10 次
2. 计算安危: 下家已立直 → 对每张候选牌计算危险度
3. 打牌排序: 按 paijia 排序，价值低的牌优先考虑打出
4. 对每张候选牌:
   a. 模拟打出 → eval_shoupai() 递归模拟后续
   b. 若危险度 > 阈值 → 跳过（除非收益极高）
5. 选出 max 期望值的牌
6. 若期望 ≥ 350 + 可立直 → 立直
```

## 六、与技能系统对接点

AI 的 `action_*` 方法可以成为技能系统介入的位置：

1. **`action_zimo`** — 摸牌后 → 可触发摸牌后技能
2. **`action_dapai`** — 打牌后 → 可触发打牌响应技能
3. **`action_fulou`** — 吃碰后 → 可触发副露响应技能
4. **`eval_shoupai`** — 评估循环可感知特殊的听牌形态

要让 AI 能使用技能，需要在 `skill/index.js` 中的 `SkillManager` 注入 AI 实例，并在 AI 的 `action_*` 方法中调用技能检查。

## 七、关键常量速查

| 常量 | 值 | 含义 |
|------|-----|------|
| `width[0]` | 8 | 听牌时模拟分支数分母 |
| `width[1]` | 32 | 一向听时模拟分支数分母 |
| `width[2]` | 256 | 二向听时模拟分支数分母 |
| `_n_zimo` 初始 | 70 | 初始剩余摸牌次数 |
| 立直危险度 | 0~100 | 庄家 ×1.5 |
| 立直门槛 | ev ≥ 350 | 期待值高于此值才立直 |
| 危险度跳过 | ≥ 13 | 绝对避免 |
